# Metadata Sidebar — Augmentations to MetadataSidebar

There is already a `MetadataSidebar` component
(`components/Sidebar/MetadataSidebar.tsx`) that renders the existing
schema for the selected resource. This document covers what it does
today and the specific augmentations the query feature requires.

Read `09-current-state.md` first.

## What MetadataSidebar already does

- Renders only when `selectedResource.type === "text"` (empty state
  for folder/image/audio).
- Iterates `schema.groups`. Skips a group when `group.folderId` is set
  and doesn't match `selectedResource.folderId`.
- Locked built-in keys get specialised controls
  (`SynopsisInput`, `NotesInput`, `StatusSelector`, `POVAutocomplete`,
  `DateTimeInput`, `DurationInput`, `EndDateInput`).
- Other fields render via generic controls keyed off `field.type`
  (`GenericTextInput`, `NumberInput`, `DateTimeInput`,
  `BooleanToggle`, `SelectInput`, `ResourceRefInput`,
  `MultiResourceRefInput`).
- `MultiResourceRefInput` already respects `refFolder` /
  `includeSubfolders` / `maxSelections`.
- `TagsSection` renders below the schema groups as a separate
  collapsible section.
- `+ Add field` button (at the bottom): dispatches `addMetadataField`
  with `{ key: "field-<timestamp>", label: "New Field", type: "text" }`
  into the first group of the schema. This declares a new field — it
  does **not** add a value to the sidecar without declaring.

## Augmentations the query feature needs

### 1. Placeholder for unset declared fields

Currently, when a declared field has no value on the resource, the
control renders empty (an empty text input, an unselected dropdown,
etc.). The field is visible but it doesn't read as "expected but
absent" — it reads as "I'm a blank input."

Add a placeholder state for unset declared fields:

```
Word goal      [ 1500 ]              ← set, shown as value
Word goal      (set goal)            ← unset, subdued, click to edit
POV            [ Mara ▾ ]            ← set
POV            (set point of view)   ← unset, subdued
```

Implementation: each control checks whether the value is
`null`/`undefined`/`""`/`[]` and renders a muted placeholder
treatment in that case. Clicking enters edit mode normally.

This is the discovery path that lets the existing schema teach
itself to writers without docs.

### 2. Improved `+ Add field` flow

The current button creates a generic text field with an autogenerated
key. Three problems:

- No autocomplete against existing field keys, so users can easily
  create duplicates (`tone`, `Tone`, `Mood`, `feel`).
- Auto-generated key (`field-<timestamp>`) is opaque and unreadable.
- Always lands in the first group, regardless of where the user wants
  it.

Replace with an inline mini-form:

```
+ Add field
─────────────
Name:  [tone__________]   ← autocomplete from all existing field keys
                            in the schema (project-wide, not just this
                            group)
Label: [Tone__________]   ← auto-derived from name; user-editable
Type:  [text ▾]           ← defaulted from value below, if entered

Optional value: [_________]

Group: [Plot ▾]           ← default = group of the autocomplete match,
                            else last-used, else first group

                                            [Cancel]  [Add]
```

**Autocomplete on name** is the single most important interaction for
keeping the field space clean. Typing `to` surfaces existing `tone`.
Selecting it dismisses the form and focuses the resource's existing
`tone` field. (No duplicate field is created.)

**Exact-name match** behaves the same as picking the autocomplete
suggestion. Pressing Enter on literal `tone` when `tone` exists
focuses the existing field, not creates a new one.

**Fuzzy match warning**: typing `Tone` when `tone` exists shows "Did
you mean `tone`?" with a one-click switch to the existing field.

**Type pre-selection from value**: if the user enters a value before
the type is chosen, infer (number / date / boolean / text) and
highlight the inferred type. The type override stays visible —
inference is a guess, not a commit.

### 3. Conflict states

The sidebar should surface three conflict modes without blocking
editing:

```
status   [ ‹deprecated› outline ▾ ]   ⚠ value no longer in options
                                          [clear] [pick new]

pov      [ ‹missing› Mara ▾ ]         ⚠ referenced resource was deleted
                                          [clear] [pick new]

tone     [ ▾ — was text, now select ] ⚠ field type changed
                                          [migrate value] [resolve in schema]
```

The first two are existing scenarios (the existing controls handle
them silently today). Surfacing them with warning state is the
augmentation.

The third only happens if the user does a type change *without*
running the migration preview from `06-fields-panel.md`. If migration
preview becomes mandatory for type changes, this state should be
rare.

### 4. Cross-link to SchemaManager

Right-clicking a field name (or hovering for a `⋮` menu) should
offer:

- **Edit in Schema Manager** — opens SchemaManager scrolled to and
  highlighting this field. Useful for renames, type changes, options
  updates that are too heavy for the sidebar.
- **Show resources with this field set** — opens a query view filtered
  to resources where this field has a non-empty value. (Requires the
  query feature.)
- **Show resources with this field empty** — same, inverted.

Most operations the writer does day-to-day stay in the sidebar; the
right-click menu is the bridge for schema-level work.

### 5. Previous-resource memory (defer to v2)

For `resource-ref` fields, surface the previous resource's value
when the writer creates a sequence of resources in the same folder:

```
POV       [ (set POV) — last: Mara ]
```

Tab to accept "last", type to override. No automatic fill — silent
defaults cause silent bugs.

Heuristic for "previous": the resource with the highest `orderIndex`
that's strictly less than the current resource's `orderIndex` in the
same `folderId`. Skip if the previous resource has the field unset.

Defer to v2 — useful, but not part of the initial query feature.

### 6. Observed fields — out of scope

The original design proposed treating ad-hoc sidecar keys as queryable
"observed" fields. `05-schema-layer.md` recommends against it for v1.
The sidebar therefore has no observed section. Every field the writer
sees is declared in `metadataSchema`.

If observed lands later, the sidebar gains a "Custom" section between
the declared groups and tags, with a small `⤴` promote affordance on
each row.
