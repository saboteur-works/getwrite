# Style Contract

This artifact captures exemplar-derived keep-rules, anti-patterns, and tie-breaker guidance for spec 004.

## Exemplar Capture Format

Use this format for each source exemplar before deriving rules.

| Exemplar ID | Source       | Why It Is Strong | Observable Pattern        | Caveats               |
| ----------- | ------------ | ---------------- | ------------------------- | --------------------- |
| EX-###      | path/to/file | short rationale  | concrete behavior/pattern | limits or scope notes |

## Rule Categories

All derived rules and anti-patterns must be categorized.

- Type safety
- Module responsibility
- State management
- Error handling
- React component shape
- Styling (token-first)
- Third-party utilities

## Exemplar Observations

### Batch 1: Model Utilities (T010)

Source files: `frontend/src/lib/models/revision-manager.ts`, `frontend/src/lib/models/resource.ts`, `frontend/src/lib/models/schemas.ts`, `frontend/src/lib/models/uuid.ts`

| Exemplar ID | Source                                        | Why It Is Strong                                                                                                                          | Observable Pattern                                                                                                              | Caveats                                                                                               |
| ----------- | --------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| EX-001a     | `frontend/src/lib/models/revision-manager.ts` | Lock acquisition is explicit and scoped to a `try/finally` block, preventing lock leaks on error paths                                    | `const release = await acquireLock(resourceId); try { ... } finally { release(); }`                                             | Pattern is only required where concurrent mutation risk exists                                        |
| EX-001b     | `frontend/src/lib/models/revision-manager.ts` | Options params with `??` defaults keep call sites readable without requiring all arguments                                                | `const maxRevisions = options.maxRevisions ?? 50;` — defaults absorbed inside the function                                      | Avoid if defaults are non-obvious; document them in JSDoc `@param`                                    |
| EX-001c     | `frontend/src/lib/models/revision-manager.ts` | Background work is fire-and-forget with explicit `void` and a swallowed-error comment, keeping the caller's error surface clean           | `void enqueueIndex(...)` inside `try { } catch (err) { // ignore queueing errors }`                                             | Only acceptable for best-effort, non-critical side effects; not for data writes                       |
| EX-001d     | `frontend/src/lib/models/revision-manager.ts` | Default export groups the public API of the module surface into a single object                                                           | `export default { nextVersionNumber, createRevision }`                                                                          | Named exports are preferred for tree-shaking; default export is the summary surface                   |
| EX-002a     | `frontend/src/lib/models/resource.ts`         | `@module` JSDoc at file top declares explicit responsibilities; readers know what belongs and what does not                               | `@module resource` block listing four bullet-point responsibilities                                                             | Must be updated when module scope changes; stale module docs are worse than none                      |
| EX-002b     | `frontend/src/lib/models/resource.ts`         | Each factory validates the constructed object through the canonical schema before returning, ensuring no invalid objects escape           | `TextResourceSchema.parse(res);` immediately before `return res;`                                                               | Schema parse re-evaluates the full object; do not skip for "simple" types                             |
| EX-002c     | `frontend/src/lib/models/resource.ts`         | Private helpers (`slugify`, type guards) are module-top and unexported; they serve the module only                                        | `function slugify(s: string): string { ... }` — no `export` keyword                                                             | Promote to shared utility only when a second caller is identified, not speculatively                  |
| EX-002d     | `frontend/src/lib/models/resource.ts`         | All factory docstrings include `@param`, `@returns`, `@throws`, and `@example`; readers can understand a factory without reading its body | `createTextResource` JSDoc with eight `@param` tags, `@returns`, `@throws`, and `@example`                                      | Example code in JSDoc must be kept in sync with the actual parameter shape                            |
| EX-002e     | `frontend/src/lib/models/resource.ts`         | `const now = new Date().toISOString()` at the top of every factory produces a consistent ISO timestamp shared by all timestamp fields     | Both `createdAt` and `updatedAt` fields reference the same `now`                                                                | Do not call `new Date()` in two places within a single factory; timestamps would diverge              |
| EX-003a     | `frontend/src/lib/models/schemas.ts`          | All Zod schemas live in one canonical module; drift between schema definition location is not possible                                    | Every runtime model schema (`TextResourceSchema`, `RevisionSchema`, etc.) is exported from this file only                       | Import schemas from here; do not inline `z.object(...)` in consuming modules                          |
| EX-003b     | `frontend/src/lib/models/schemas.ts`          | Schema inheritance via `.extend()` keeps the base contract in one place, and subtypes add only what differentiates them                   | `TextResourceSchema = ResourceBaseSchema.extend({ type: z.literal("text"), ... })`                                              | `.extend()` produces a new schema; mutations to the child do not back-propagate to the base           |
| EX-003c     | `frontend/src/lib/models/schemas.ts`          | `z.lazy()` is used for recursive schemas; no workaround types or casts required                                                           | `const TipTapNodeSchema: z.ZodTypeAny = z.lazy(() => z.object({ ..., content: z.array(TipTapNodeSchema).optional() }))`         | Explicit `: z.ZodTypeAny` type annotation is required to prevent TypeScript circular-inference errors |
| EX-003d     | `frontend/src/lib/models/schemas.ts`          | The `Schemas` convenience bundle is exported alongside named exports to aid discoverability in tooling                                    | `export const Schemas = { UUID, MetadataValue, ProjectSchema, ... }`                                                            | Prefer named imports in production code for tree-shaking; bundle is for REPL or exploratory use       |
| EX-004a     | `frontend/src/lib/models/uuid.ts`             | Graceful-degradation chain (`randomUUID` → `getRandomValues` → `Math.random`) means the utility works across all JS environments          | `if (g.crypto && typeof g.crypto.randomUUID === "function") { return g.crypto.randomUUID(); }` followed by byte-array fallbacks | Math.random fallback is not cryptographically secure; upgrade if security context ever changes        |
| EX-004b     | `frontend/src/lib/models/uuid.ts`             | Safe narrowing of `globalThis` without `any` using `as unknown as { crypto?: GlobalCrypto }` keeps the type surface explicit              | `const g = globalThis as unknown as { crypto?: GlobalCrypto };`                                                                 | This pattern is for well-known global shapes only; avoid for arbitrary runtime feature detection      |
| EX-004c     | `frontend/src/lib/models/uuid.ts`             | File is single-purpose: UUID generation and validation, nothing else; adding unrelated helpers here would violate module responsibility   | Only `generateUUID` and `isValidUUID` are exported from `uuid.ts`                                                               | Do not expand this file; create a new module for non-UUID identity utilities                          |

| EX-008c     | `frontend/src/lib/projectTypes.ts`                         | Named constants for both primary and fallback directory paths prevent duplication and make path changes a one-line edit                                                                                  | `const DEFAULT_TEMPLATES_DIR = path.join(...)` and `const ALT_TEMPLATES_DIR = path.join(...)`                                                                          | Do not inline path strings; string duplication across a module makes path governance impossible                                                                          |
| EX-008d     | `frontend/src/lib/projectTypes.ts`                         | Invalid-file resilience: JSON parse and validation errors are caught per file and the file is skipped, not a thrown error that halts loading                                                             | `try { const raw = await readFile(fp, "utf8"); ... } catch (_) { /* skip */ }` in the per-file loop                                                                    | Log skipped files in debug/verbose mode; silent skipping makes misconfigured templates invisible in production                                                           |
| EX-009a     | `frontend/src/lib/toast-service.ts`                        | Object-as-service pattern groups all toast methods under one exported constant with semantic names, preventing ad-hoc `toast.success(...)` calls throughout the codebase                                 | `export const toastService = { success, error, loading, info, dismiss, dismissAll }`                                                                                   | Do not call `toast` from `react-hot-toast` directly in components; use `toastService` to ensure consistent config                                                        |
| EX-009b     | `frontend/src/lib/toast-service.ts`                        | Toast configuration (`position: "bottom-right"`, duration) is centralized inside the service; no consumer needs to specify it                                                                            | Every method passes `{ position: "bottom-right", duration: ... }` from within `toastService`                                                                           | Changing position or duration requires one-location edit; do not override config at call sites                                                                           |
| EX-009c     | `frontend/src/lib/toast-service.ts`                        | Optional `description` parameter is composed into a single string inside the method; callers are not responsible for formatting                                                                          | `const content = description ? \`${message}\n${description}\` : message;`                                                                                              | Toast library renders this as a single string; if richer layout (HTML/JSX) is needed, the service must evolve, not individual call sites                                 |

| EX-010a     | `frontend/components/Sidebar/controls/MultiSelectList.tsx` | Exported props interface (`MultiSelectListProps`) sits beside the component; consumers can import the type without importing the component                                                               | `export interface MultiSelectListProps { items: string[]; selected?: string[]; ... }`                                                                                  | Props interfaces must be exported if used in parent components or wrappers                                                                                               |
| EX-010b     | `frontend/components/Sidebar/controls/MultiSelectList.tsx` | Controlled-equivalent pattern: internal `useState` is seeded from the prop, then synced via `useEffect` when the prop changes, keeping the component usable in both controlled and uncontrolled contexts | `const [sel, setSel] = React.useState<string[]>(selected); React.useEffect(() => setSel(selected), [selected]);`                                                       | This pattern creates temporary divergence between internal and prop state on rapid prop changes; prefer fully controlled if the parent manages selection authoritatively |

| EX-011b     | `frontend/components/common/ConfirmDialog.tsx`             | Focus management on open uses `setTimeout(..., 50)` to defer until the DOM is stable after the render cycle                                                                                              | `setTimeout(() => confirmRef.current?.focus(), 50);` inside the `isOpen` branch                                                                                        | The 50ms magic number is empirical; if focus fails on slow machines, consider `requestAnimationFrame` or a focus-visible polyfill                                        |

## Keep-Rules

| Rule ID | Category              | Rule                                                                                                                                                                                                                     | Source Exemplars       | When to Apply                                                                                              |
| ------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------- | ---------------------------------------------------------------------------------------------------------- |
| SR-001  | Type safety           | Call `Schema.parse(value)` immediately before every factory `return` statement; never return an unvalidated object                                                                                                       | EX-002b                | When returning newly constructed model objects from any factory function                                   |
| SR-002  | Type safety           | Import all runtime Zod schemas from `schemas.ts`; do not inline `z.object(...)` in consuming modules                                                                                                                     | EX-003a                | Any time a runtime validator is needed outside the schemas module                                          |
| SR-003  | Type safety           | Extend subtypes from a shared base schema using `.extend()`; do not copy or repeat base fields                                                                                                                           | EX-003b                | When defining a model subtype that shares fields with an existing base schema                              |
| SR-004  | Type safety           | Declare recursive Zod schemas with `z.lazy()` and an explicit `: z.ZodTypeAny` type annotation                                                                                                                           | EX-003c                | When a schema's fields reference the defining schema itself                                                |
| SR-005  | Type safety           | Narrow runtime globals with `as unknown as { prop?: Type }` rather than `as any`                                                                                                                                         | EX-004b                | When accessing well-known global shapes that TypeScript cannot infer from declarations                     |
| SR-006  | Type safety           | Derive `RootState`, `AppDispatch`, and `AppStore` types from the store factory return type; never hand-write Redux type aliases                                                                                          | EX-006b                | When adding or modifying Redux store configuration                                                         |
| SR-007  | Module responsibility | Add a `@module` JSDoc block to every non-trivial lib or service file listing its explicit responsibilities and I/O contract                                                                                              | EX-002a, EX-008a       | When creating or substantially modifying a lib, service, or utility module                                 |
| SR-008  | Module responsibility | Keep internal helpers unexported; promote a helper to a shared utility only after a second distinct caller is identified                                                                                                 | EX-002c, EX-004c       | When writing helper functions that serve only the current module                                           |
| SR-009  | Module responsibility | Name file-system path strings as module-level constants; do not inline path literals in function bodies                                                                                                                  | EX-008c                | When a module reads from or writes to any file-system location                                             |
| SR-010  | Module responsibility | Keep files single-purpose; create a new module rather than expanding an existing file beyond its declared `@module` scope                                                                                                | EX-004c                | When an addition would require adding a new responsibility bullet to the file's `@module` block            |
| SR-011  | Module responsibility | Group all public operations of a service or utility layer under one exported constant; do not scatter independent named exports across call sites                                                                        | EX-001d, EX-009a       | When writing a service or manager module with multiple related operations                                  |
| SR-012  | State management      | Declare a typed state interface at the top of each slice file; do not use anonymous object literals or inferred state shapes                                                                                             | EX-005a                | When creating or modifying a Redux slice                                                                   |
| SR-013  | State management      | Use `Map` for batch update reducers where payload size is non-trivial or unbounded; use `findIndex` only for single-item updates                                                                                         | EX-005b                | When a reducer applies an array of partial updates to a collection                                         |
| SR-014  | State management      | Use `createSelector` for derived state that spans multiple slice arrays; raw selectors are sufficient for single-array lookups                                                                                           | EX-005c                | When a selector must compose data from more than one stateful collection                                   |
| SR-015  | State management      | Pick one reducer mutation style per slice — explicit `return state` or implicit Immer mutation — and apply it to every reducer in that slice                                                                             | EX-005d                | When writing or reviewing reducer functions inside a slice                                                 |
| SR-016  | State management      | Co-locate all selectors and `createAsyncThunk` definitions in the same file as their slice                                                                                                                               | EX-005e                | When adding async actions or derived selectors for a state slice                                           |
| SR-017  | State management      | Use `makeStore` as the production store API; annotate the legacy singleton `store` with `@deprecated` until all consumers migrate                                                                                        | EX-006a                | When consuming or configuring the Redux store                                                              |
| SR-018  | State management      | Access Redux state and dispatch exclusively through typed hooks (`useAppSelector`, `useAppDispatch`) from `hooks.ts`                                                                                                     | EX-007a, EX-007b       | In any React component that reads from or dispatches to the store                                          |
| SR-019  | Error handling        | Acquire locks inside a `try/finally` block; always call the release function in `finally`, never in the `try` body                                                                                                       | EX-001a                | When using any mutual-exclusion or resource-locking primitive                                              |
| SR-020  | Error handling        | Absorb option defaults with `??` inside the function body; do not require call sites to pass all optional configuration fields                                                                                           | EX-001b                | When writing functions that accept an optional configuration parameter object                              |
| SR-021  | Error handling        | Mark fire-and-forget side effects with explicit `void`; swallow errors only with an inline comment explaining why they are safe to ignore                                                                                | EX-001c                | When intentionally not awaiting a non-critical background operation                                        |
| SR-022  | Error handling        | Isolate per-item errors in collection-loading loops; skip the invalid item and continue rather than throwing and aborting the load                                                                                       | EX-008d                | When iterating over user-provided or file-system-sourced items in a loading function                       |
| SR-023  | Error handling        | Capture the return value of `toastService.loading()`; always dismiss the loading toast after async resolution or failure                                                                                                 | EX-009d                | When using the loading toast pattern for async operations                                                  |
| SR-024  | Error handling        | Capture `const now = new Date().toISOString()` once per factory call and reference it for all timestamp fields in the object                                                                                             | EX-002e                | When constructing objects with more than one timestamp field                                               |
| SR-025  | React component shape | Export the props interface alongside each component file so consumers can import the type independently of the component                                                                                                 | EX-010a                | When defining any named component with a non-trivial prop surface                                          |
| SR-026  | React component shape | Register document-level event listeners inside `useEffect` with a cleanup return that removes the same listener; never add global listeners outside effects                                                              | EX-011a                | When a component requires keyboard or pointer event handling at document level                             |
| SR-027  | React component shape | Apply the full ARIA dialog contract (`role="dialog"`, `aria-modal="true"`, `aria-labelledby`) to modal overlays and implement a keyboard focus trap                                                                      | EX-011c                | When building any blocking overlay, confirmation panel, or modal dialog                                    |
| SR-028  | React component shape | Guard conditional component render with an early `return null` before the JSX block rather than a top-level conditional JSX wrapper                                                                                      | EX-011e                | When a component that is conditionally visible has non-trivial render output                               |
| SR-029  | React component shape | Document every exported factory, service method, and utility function with `@param`, `@returns`, `@throws`, and `@example`                                                                                               | EX-002d                | When writing or modifying any exported function whose behavior cannot be inferred from its signature alone |
| SR-030  | Styling (token-first) | Use design-system token-based class names for all visual styling on application components; do not apply raw Tailwind utility classes in component JSX                                                                   | EX-011d vs EX-010c     | When adding or modifying any styles on an application component                                            |
| SR-031  | Styling (token-first) | Use semantic BEM-style class names for complex interactive components (dialogs, overlays, menus) and centralize their styles in a stylesheet                                                                             | EX-011d                | When building interactive components with more than one internal visual zone                               |
| SR-032  | Third-party utilities | Use lodash only for utilities with no idiomatic native equivalent (`debounce`, `throttle`, `cloneDeep`, `uniq`, `groupBy`); do not import lodash equivalents of native array methods (`map`, `find`, `filter`, `reduce`) | — (see AP-020)         | When reaching for a collection or timing utility; check whether the native method covers the need first    |
| SR-033  | Third-party utilities | When importing from lodash, use path imports (`import debounce from 'lodash/debounce'`) or `lodash-es` named imports; never use `import { fn } from 'lodash'` (full-package CJS import does not tree-shake)              | — (see AP-020, AP-021) | Any time a lodash function is imported                                                                     |

## Anti-Patterns

| Anti-Pattern ID | Category              | Avoid This                                                                                                                | Why It Causes Drift                                                                                                                                  | Better Direction                  |
| --------------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------- |
| AP-001          | Type safety           | Returning a newly constructed model object without calling `Schema.parse()`                                               | Invalid shapes escape the factory silently and cause unpredictable downstream failures                                                               | SR-001                            |
| AP-002          | Type safety           | Inlining `z.object(...)` validators inside consuming modules                                                              | Schema definitions fragment from the canonical source in `schemas.ts`; updating a shape requires hunting all call sites                              | SR-002                            |
| AP-003          | Type safety           | Casting unknown values or globals with `as any`                                                                           | Eliminates TypeScript's static checks; any property access succeeds silently at the cost of runtime safety                                           | SR-005                            |
| AP-004          | Type safety           | Hand-writing `RootState`, `AppDispatch`, or `AppStore` type aliases                                                       | Aliases fall out of sync when reducers or middleware change; type errors surface at runtime instead of compile time                                  | SR-006                            |
| AP-005          | Module responsibility | Exporting a helper that is only used within its defining module                                                           | Artificially enlarges the public API surface, increases coupling, and makes the module boundary misleading                                           | SR-008                            |
| AP-006          | Module responsibility | Inlining file-system path strings as literals in a function body                                                          | Path changes require grep-and-replace; string duplication across a module makes governance impossible                                                | SR-009                            |
| AP-007          | Module responsibility | Adding unrelated operations to an existing module rather than creating a new one                                          | Violates single-purpose design; `@module` block becomes inaccurate and the file grows without a clear boundary                                       | SR-010                            |
| AP-008          | State management      | Deriving cross-list state in a component body (e.g., combining folders and resources with `Array.concat` inside JSX)      | Derivation re-runs on every render; the derived value cannot be shared or memoized across components                                                 | SR-014                            |
| AP-009          | State management      | Mixing implicit Immer mutation with explicit `return state` in the same slice                                             | Mutation intent becomes ambiguous; bugs from accidental re-assignment are hard to trace during code review                                           | SR-015                            |
| AP-010          | State management      | Defining selectors inline in `useSelector` calls or inside component files                                                | Selectors cannot be reused or independently memoized; slice data contract leaks into the UI layer                                                    | SR-016                            |
| AP-011          | State management      | Importing `useSelector` or `useDispatch` from `react-redux` directly in components                                        | Bypasses typed hooks; state reads and dispatches lose compile-time type guarantees                                                                   | SR-018                            |
| AP-012          | Error handling        | Awaiting a lock-protected operation with the release call inside the `try` body rather than `finally`                     | The lock is not released when an exception is thrown; subsequent operations that need the lock will deadlock                                         | SR-019                            |
| AP-013          | Error handling        | Omitting `void` on a fire-and-forget promise or swallowing an error without a comment                                     | Unhandled promise rejections pass silently; linting and runtime environments may surface intermittent warnings                                       | SR-021                            |
| AP-014          | Error handling        | Letting a single invalid item in a collection-loading loop throw and abort the entire load                                | All valid items remain unloaded due to one bad file; the user sees an empty state with no signal                                                     | SR-022                            |
| AP-015          | Error handling        | Calling `new Date()` twice within a single factory to set `createdAt` and `updatedAt`                                     | Timestamps diverge by milliseconds; the inconsistency corrupts audit logs and sort-by-date logic                                                     | SR-024                            |
| AP-016          | React component shape | Adding a document-level event listener outside a `useEffect` or without a cleanup return                                  | Listeners accumulate across mounts and navigations; causes duplicate keyboard or focus handling bugs                                                 | SR-026                            |
| AP-017          | React component shape | Using `isOpen && <Modal />` conditional JSX instead of an early-return guard for non-trivial panels                       | Control flow is harder to follow; ARIA attributes and focus behavior depend on the component being fully unmounted when closed                       | SR-028                            |
| AP-018          | React component shape | Omitting `aria-modal`, `aria-labelledby`, or a keyboard focus trap from a blocking overlay                                | Screen-reader users cannot navigate or dismiss the dialog; WCAG 2.1 criterion 2.1.1 and 4.1.2 fail                                                   | SR-027                            |
| AP-019          | Styling (token-first) | Applying raw Tailwind utility classes (e.g., `text-sm`, `font-medium`, `mt-2`) directly on application component elements | Visual output diverges from design-system token values; refactoring the design system requires hunting individual class names across components      | SR-030 (see tie-breaker guidance) |
| AP-020          | Third-party utilities | Importing `map`, `find`, `filter`, or `reduce` from lodash when the input is a plain array                                | These are behaviorally identical to native methods on arrays; the import adds bundle weight and weakens TypeScript inference with no functional gain | SR-032                            |
| AP-021          | Third-party utilities | Using `import { fn } from 'lodash'` (full-package named import) for any lodash function                                   | The CJS `lodash` package does not tree-shake; the entire ~70 KB library is included in the bundle chunk regardless of which functions are used       | SR-033                            |

## Tie-Breaker Guidance

This section resolves cases where two or more exemplars or rules point in different directions. Use these as decision rules in priority order: apply the first rule whose condition matches and stop.

---

### TB-001 — Product invariant over stylistic preference

**Condition**: A refactor choice would simplify code but would alter or weaken a product invariant (Workspace behavior, revision/canonical integrity, resource identity, compile non-mutation, or token-first styling contract).

**Resolution**: Reject the simplification. Product invariants are non-negotiable hard gates regardless of how clean the alternative looks. Specifically:

- Do not collapse or merge code paths that enforce canonical revision selection.
- Do not remove validation calls (e.g., `Schema.parse()`) on the grounds that the upstream is "already trusted."
- Do not replace token-backed class names with raw utility values even when the visual result appears identical.

**Rationale**: Research decision "Preserve product and design-system constraints as hard gates" explicitly classifies these as authoritative; no style preference overrides them.

---

### TB-002 — Repository exemplar over ecosystem convention

**Condition**: An ecosystem convention conflicts with an observable pattern from a named exemplar (EX-###) in this contract.

**Resolution**: Follow the repository exemplar. The research decision "Derive target style from exemplar code already in the repository" establishes that repository-grounded patterns take precedence over generic framework best practices.

**Common conflict scenarios**:

- A Redux guide suggests inline selectors; the repository uses co-located selectors per the `resourcesSlice.ts` exemplar (EX-005e) → follow SR-016, use co-located selectors.
- An upstream library example uses `useSelector` directly; the repository wraps it in `useAppSelector` (EX-007a) → follow SR-018, use the typed hook.
- A library tutorial returns an unvalidated object from a builder; the repository always calls `Schema.parse()` (EX-002b) → follow SR-001, validate before returning.

---

### TB-003 — Token-first styling when tooling examples conflict

**Condition**: A component migration could retain existing Tailwind utility classes (as in EX-010c) or adopt semantic token-backed class names (as in EX-011d), and it is unclear which to apply.

**Resolution**: Always choose token-backed semantic class names for application components. Apply in this order:

1. **Reuse existing token-backed utility classes** from `frontend/styles/tokens.css` and `frontend/styles/utilities.css` — prefer `project-modal-*`, `workarea-*`, `appshell-*`, and similar.
2. **Use CSS custom properties** (`var(--color-*)`, `var(--space-*)`, `var(--radius-*)`) directly in a scoped stylesheet when no matching utility exists.
3. **Introduce a new token-backed utility class** in `frontend/styles/utilities.css` if the pattern will be reused across two or more components.
4. **Do not introduce hard-coded values** (hex colors, pixel sizes, shadow strings) at any step.

Raw Tailwind utilities (`text-sm`, `font-medium`, `mt-2`, etc.) are only acceptable in prototype or throwaway code. Any component entering the production codebase must not contain raw Tailwind utilities on host elements (inputs, buttons, containers that carry brand/layout intent).

**Authority**: Tome `tome-004-token-first-component-styling-preference` confirms this as a standing implementation preference and cites `frontend/styles/tokens.css` and `frontend/styles/utilities.css` as the canonical sources.

---

### TB-004 — Simple interface over consistency when safety is not at stake

**Condition**: Two exemplars demonstrate different approaches (e.g., controlled vs. controlled-equivalent state management in EX-010b), and neither involves a product invariant or type-safety concern.

**Resolution**: Prefer the approach that reduces props burden on the parent. Specifically:

- If the parent manages selection state authoritatively, use a **fully controlled** pattern (prop drives all state; no internal `useState`).
- If the parent only sets an initial value and does not need to reactively update it, use the **controlled-equivalent** pattern (EX-010b: seed `useState` from prop, sync with `useEffect`).
- Do not use controlled-equivalent when the parent will change the prop value frequently — the intermediate divergence creates bugs.

When both approaches are equally defensible after this test, follow the pattern already used in the nearest sibling component in the same directory.

---

### TB-005 — Error resilience over DRY in loading functions

**Condition**: A refactor could DRY up a collection-loading loop by extracting repeated logic, but the extraction would merge per-item error handling into a single shared catch block that aborts on the first failure.

**Resolution**: Preserve per-item isolation (SR-022). Do not extract per-item try/catch into a shared handler if the shared handler would throw and halt the loop. Prefer explicit duplication of the error boundary over a DRY abstraction that changes the load-failure behavior.

**Rationale**: Silent per-item skipping is a deliberate resilience choice (EX-008d). The user sees a partial result instead of an empty state, which is always preferable for file-system-sourced collections.

---

### TB-006 — Explicit cleanup over terseness in effect hooks

**Condition**: A `useEffect` registers a side effect (event listener, timer, subscription) and it is tempting to skip the cleanup return because "the component unmounts rarely."

**Resolution**: Always return a cleanup function. No effect that registers a global or document-level side effect is exempt. This is not a style choice — missing cleanups compound across navigation events and produce hard-to-reproduce bugs (AP-016 / SR-026).

If a cleanup truly does nothing (e.g., a fire-and-forget fetch with no subscription), write an explicit `return () => {}` with a comment explaining why no cleanup is required. The audit trail is more valuable than the saved lines.

### TB-007 — Lodash utility vs. native method

**Condition**: A piece of code could use a lodash collection function (e.g., `_.map`, `_.find`, `_.filter`) or the equivalent native `Array.prototype` method, and it is unclear which to apply.

**Resolution**: Use the native method unless lodash provides genuinely distinct behavior that the native method cannot replicate without non-trivial additional code. Specifically:

- **Prefer native**: `array.map()`, `array.find()`, `array.filter()`, `array.reduce()`, `array.some()`, `array.every()` — these are identical to lodash equivalents on plain arrays and require no import.
- **Use lodash**: `debounce`, `throttle`, `cloneDeep`, `uniq`, `uniqBy`, `groupBy`, `keyBy`, `chunk`, `difference`, `intersection`, `merge` — these either have no native equivalent or their native approximation requires significant extra code.
- **Migration direction**: existing calls to `_.filter(array, predicate)` or `_.map(array, fn)` in components such as `OrganizerView.tsx` should be replaced with `array.filter(predicate)` and `array.map(fn)` when touched during refactor.

**Import rule when lodash is legitimately needed**: use path imports or `lodash-es`, never the full-package named import (see SR-033, AP-021). Existing usages of `import debounce from 'lodash/debounce'` in `AppShell.tsx` and `EditView.tsx` are already conformant and are the reference pattern.

**Authority**: SR-032, SR-033, AP-020, AP-021; tome `tome-007-lodash-usage-policy`.
